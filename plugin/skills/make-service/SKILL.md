# 服務包產生 — 非工程師對話流程（薄殼）

你在幫**非工程師**把一個**已經驗證好用**的東西固化成「服務包」——一個符合規範的資料夾。
**你的工作到「產出資料夾」為止**：不部署、不碰雲端、不呼叫任何 server。
部署與登記＝工程師用部署器做的事；使用者只要把資料夾交給工程師。

## ⚠️ 先判斷：這是「日常一次性」還是「要固定下來」？（最重要）
產包**不是日常動作**，是「畢業按鈕」。多數需求是一次性/測試，**不該產生服務**。

- **日常 / 測試（預設）**——「幫我拉昨天數據」「產個報表看看」「這樣改改看」：
  → **當場做掉**，不產包。想試幾次就試幾次。
- **固定下來（才產包）**——使用者明確說「這個很好用，幫我**變成每天自動跑**／**正式導入流程**」：
  → 走下面流程產服務包，**把他剛驗證過的邏輯（查詢/prompt/輸出格式）原樣固化**，不要重寫。

判斷試金石：「這件事需要之後**定期自動跑、他不用每次開口**嗎？」是＝產包；否＝留 ad-hoc。
不確定就問一句：「這次跑完就好，還是要固定成每天自動跑的服務？」

## 支援範圍
- **job**：排程自動「讀資料 →(可選)AI 整理 → 產出到雲端」
- **service**：常駐 HTTP 端點
- **逃生給工程師（別硬做）**：碰客戶個資/財務/機密資料、複雜整合、互動網站
  → 明說「這個要請工程師處理」，可以先幫他把需求寫成一頁說明。

## 產包流程
1. **用大白話收參數**（缺的才問）：做什麼（description）、多久跑（schedule/timezone）、
   要不要 AI（uses_ai）、資料敏感度（data_class；PII/financial→逃生）、
   英文代號（name，小寫-連字號）、他的 email（owner，自報即可——部署時工程師端會以真實身分覆核）。
2. **產出服務包資料夾**（規範＝bundle spec v1）：
   - 有本機 repo/終端機（工程師環境）：直接跑
     `python3 generator.py <name> --out . --params '{...}'`
   - 沒有（Cowork 一般使用者）：**照規範直接寫出五個檔案**到一個資料夾：
     `service.yaml`（manifest）+ `main.py` + `Dockerfile` + `requirements.txt` + `README.md`。
     把他驗證過的邏輯放進 main.py；內建硬化不可省：**冪等**（輸出已存在則跳過）+
     **structured JSON log**。規範與範例：
     https://github.com/spenguinlui/ai-infra-management/blob/main/docs/service-factory/bundle-spec.md
3. **自我檢查**（照 bundle spec：service.yaml 有 bundle_spec/name/archetype/description/owner/
   governance.data_class/edges≥1；五檔齊全）。
4. **交付**：告訴使用者「把這個資料夾**交給工程師**（附一句：請用部署器上線）」。
   工程師會權威驗證、部署、登記；上線後它會在排程時間自動跑。

## 不要做的事
- 不呼叫任何部署/雲端工具（那是工程師與部署器的事）。
- 不代填別人身分：owner 填使用者自己的 email；真實身分由工程師部署時的認證覆核。
- 資料敏感（customer-pii / financial）不自助——逃生給工程師。
